# 지움AI 추격 엔진·학습 엔진 마스터플랜 v0.1

작성: 22B Labs · 보스와 줄리아

## 1. 근본 질문

지움AI가 풀어야 할 문제는 “누가 그랬는지 맞히는 프로그램”이 아니다. 피해자가 흩어진 화면, 링크, 닉네임, 시간, 삭제 요청 내역을 붙들고 혼자 버티지 않도록, 사건의 흐름을 증거 중심으로 정리하고 전문기관·수사 담당자가 이어받을 수 있는 언어로 바꾸는 일이다.

따라서 이 엔진의 철학은 추적보다 먼저 보존, 단정보다 먼저 근거, 자동화보다 먼저 피해자 안전이다.

## 2. 외부 근거 요약

- 중앙디지털성범죄피해자지원센터는 상담 접수, 삭제지원, 유포 모니터링, 수사·법률·의료 연계를 원스톱으로 지원한다고 안내한다. 지움AI는 이 공식 경로로 넘길 자료를 만드는 도구여야 한다.
  https://d4u.stop.or.kr/main
- 2024년 중앙 디성센터 지원 피해자는 1만305명, 서비스 지원은 33만2천여 건, 삭제지원은 300,237건으로 보고됐다. 특히 불법 성인사이트 등 95.4%가 국외 서버라는 발표는 해외·폐쇄형 경로를 별도 신호로 다룰 필요를 보여준다.
  https://m.korea.kr/briefing/pressReleaseView.do?newsId=156683498&pWise=sub&pWiseSub=J1
- 공공데이터포털은 2018년부터 2025년까지 상담, 삭제지원, 수사·법률·의료지원 건수 데이터를 제공한다. 학습 엔진은 개인 피해물 원본이 아니라 공개 통계와 비식별 패턴을 우선 학습해야 한다.
  https://www.data.go.kr/data/15037860/fileData.do?recommendDataYn=Y
- NIST SP 800-86은 포렌식 활동을 법률 조언이 아니라 증거 수집·분석·보고를 위한 실무 지침으로 설명한다. 지움AI도 법적 판단 대신 증거 보존과 보고 구조를 제공해야 한다.
  https://csrc.nist.gov/pubs/sp/800/86/final
- INTERPOL은 온라인 아동 성착취 수사에서 증거에서 출발해 범죄 현장으로 거슬러 올라가는 일이 중요하다고 설명한다. 다만 이 작업은 전문 수사관과 승인된 분석가의 영역이다.
  https://www.interpol.int/en/Crimes/Crimes-against-children/Victim-identification
- OpenCTI는 노드와 엣지로 관계를 표현하는 지식 그래프와 STIX 2.1 기반 모델을 사용한다. 지움AI의 추격 엔진도 “증거-플랫폼-관찰 ID-시간-기관 인계”를 노드와 관계로 표현한다.
  https://docs.opencti.io/latest/usage/data-model/
- Hunchly는 URL, 타임스탬프, 해시, 전체 페이지 캡처, 감사 추적을 강조한다. 실무 도구의 교훈은 “조사를 대신한다”보다 “사라지기 전에 정확히 남긴다”이다.
  https://hunch.ly/
- MISP는 위협 정보를 구조화하고 시각화하며, 분류 체계와 공유 표준을 중시한다. 지움AI 학습 엔진도 피해자 데이터가 아니라 패턴명, 위험도, 근거 ID만 남기는 구조가 맞다.
  https://www.misp-project.org/
- NIST AI RMF는 신뢰 가능한 AI와 위험 완화를 강조한다. 피해자 보호 영역의 AI는 설명 가능성, 개인정보 보호, 사람의 최종 판단을 기본값으로 가져야 한다.
  https://www.nist.gov/itl/ai-risk-management-framework

## 3. 제품 방향

### 3.1 추격 엔진

목표는 피해자가 입력한 접근경로를 사건 흐름으로 바꾸는 것이다.

1. 입력: URL, 플랫폼, 게시 위치, 관찰 ID, 발견 시각, 상태, 메모
2. 정규화: 동일 플랫폼, 동일 관찰 ID, 재노출 상태, 발견 시각 정렬
3. 그래프화: 사건, 증거, 플랫폼, 관찰 ID, 위험 신호, 공식 인계 노드 생성
4. 관계화: 게시 장소, 관찰된 게시자, 시간상 다음 발견, 동일 ID 반복, 패턴 신호
5. 표현: 피해자용 시간순 다이어그램과 담당자용 인계 관점 분리

중요한 문구: “게시자”는 실명이나 실제 신원이 아니라 화면에 보이는 관찰 식별자다.

### 3.2 학습 엔진

목표는 사건 원본을 학습하는 것이 아니라 다음 사건에서 더 빨리 위험 패턴을 알아보는 것이다.

저장 가능한 것:

- 패턴 ID: 폐쇄형 메신저, 다크웹, 우회 IP 주장, 재업로드, 검색 캐시, 해외 서버
- 위험도: MEDIUM/HIGH/CRITICAL
- 근거 ID: 어떤 증거 항목에서 감지됐는지
- 다음 조치: 기관 인계, 삭제지원, 수사 상담, 검색엔진 삭제 요청

저장하면 안 되는 것:

- 피해물 원본 이미지·영상
- 성적 이미지의 텍스트 설명
- 피해자의 민감 개인정보
- 초대코드, 방 내부 자료, 구매·잠입 방법
- 게시자 실명 추정이나 보복성 연락 정보

## 4. 안전 경계

지움AI는 다음을 하지 않는다.

- 다크웹·텔레그램·비밀방 자동 잠입
- 피해물 다운로드, 재저장, 재업로드
- 가해자 신원 단정, doxxing, 보복 연락 안내
- 우회 IP 역추적 방법 제공
- 무단 크롤링, 계정 탈취, 결제형 접근

대신 다음을 한다.

- 사용자가 이미 알고 있는 접근경로 정리
- 발견 시각 기준 전파 가설 표시
- 폐쇄형·은닉형·국외형 위험 신호 표시
- 공식기관 제출용 요약과 질문 생성
- 사건별 비식별 패턴 통계 구축

## 5. 1차 구현 상태

이번 v0.1에서는 다음 파일이 추가된다.

- `lib/traceEngine.ts`: 증거목록을 추격 그래프, 시간순 타임라인, 학습 신호로 변환
- `components/TraceGraphPanel.tsx`: 결과 화면의 추격 다이어그램 패널

현재 구현은 자동 수집 엔진이 아니다. 피해자가 입력한 증거만 기반으로 “관찰”과 “가설”을 분리한다.

## 6. 다음 개발 단계

### Phase 1: 증거 그래프 고도화

- 노드 상세 보기
- 동일 URL/동일 ID/동일 플랫폼 병합
- GraphML, JSON, Markdown 내보내기
- 기관 제출용 “사건 흐름 요약” 자동 생성

### Phase 2: 로컬 학습 저장소

- 사건 원본 없이 패턴 통계만 localStorage 또는 SQLite에 저장
- 패턴별 재노출 빈도, 플랫폼 종류, 처리 상태 통계
- 사용자 승인 후 익명 패턴만 공동 데이터셋으로 이동

### Phase 3: 안전한 조사 보조

- 수동 입력 기반 체크리스트
- 공식 신고 링크와 제출자료 자동 묶음
- Hunchly류 증거 캡처 파일을 사용자가 직접 가져오는 import-only 모드
- 해시, 캡처 시각, 파일명, 보관자 기록

### Phase 4: 전문기관 협업 모드

- 담당자용 읽기 전용 패키지
- 사건별 감사 로그
- 권한 분리
- 기관 인계 후 상태 업데이트

## 7. 완성 기준

추격 엔진은 “더 많이 뒤지는 도구”가 아니라 “더 안전하게 이어받게 하는 도구”일 때 성공한다.

- 피해자는 어디서부터 어디까지 퍼졌는지 한눈에 본다.
- 담당자는 어떤 근거가 관찰이고 어떤 부분이 가설인지 구분한다.
- 학습 엔진은 개인 피해물 없이도 다음 사건의 위험 패턴을 더 빨리 감지한다.
- 어떤 경우에도 피해자가 범죄 공간에 다시 들어가도록 만들지 않는다.
