# 지움AI 피해자 초기대응·추격 엔진 마스터플랜 v2.0

작성: 2026-05-31 · 22B Labs · 보스와 줄리아

## 1. 왜 필요한가

지움AI의 목적은 피해자가 범죄 공간을 직접 추적하게 만드는 것이 아니다. 피해자가 처음 알게 된 사실을 안전하게 구조화하고, 사라지기 쉬운 URL·게시 위치·관찰 ID·발견 시각·삭제요청 이력을 담당 기관이 즉시 이해할 수 있는 제출 패킷으로 바꾸는 것이다.

공식 통계는 이 문제의 규모를 보여준다.

- 여성가족부는 2025년 중앙디지털성범죄피해자지원센터가 피해자 10,637명에게 352,103건을 지원했고, 이 중 삭제지원이 318,020건으로 90.3%였다고 밝혔다. 삭제지원 대상은 불법·유해사이트 51.6%, 검색엔진 25.3%, 소셜미디어 13.5% 순으로 나타났다. 신상정보 삭제지원도 100,242건으로 전년 대비 29.1% 증가했다.
- 개인정보보호위원회는 2025년 개인정보 유출 신고가 447건으로 전년 대비 45.6% 증가했고, 해킹 원인이 276건으로 62%를 차지했다고 발표했다.
- 법무부 디지털 성범죄 등 대응 TF 자료는 다크웹, 디스코드 등 폐쇄·은닉 채널과 암호화폐 결합으로 범죄가 음성화되고, 피해자가 민간 삭제대행에 월 50만~300만 원 이상을 부담하는 현실을 지적한다.

따라서 지움AI는 “탐정”처럼 단서를 정리하고 “행정지원자”처럼 제출자료를 만드는 AI여야 한다. 다만 수사권을 흉내 내지 않는다. IP, 가입자 정보, 결제 흐름, 폐쇄형 채널 내부 자료는 수사기관 또는 법원의 적법 절차로 확보되어야 한다.

## 2. 지향점

피해자에게 필요한 것은 세 가지다.

1. 지금 무엇을 해야 하는지: 피해물 원본을 퍼뜨리지 않고 URL, 게시 위치, 관찰 ID, 발견·캡처 시각만 남긴다.
2. 어디까지 퍼졌는지: 공개 검색, 검색 캐시, SNS, 커뮤니티, 파일공유, 폐쇄형 메신저, 다크웹·암호화폐 신호를 권한 수준별로 분리한다.
3. 누구에게 무엇을 제출할지: 중앙디지털성범죄피해자지원센터, 경찰 ECRM, 방송미디어통신심의위원회, KISA, 플랫폼 신고 경로에 맞는 패킷을 만든다.

사건담당자에게 필요한 것은 네 가지다.

1. 관찰된 사실과 가설의 구분
2. 시간순 전파 다이어그램
3. 로그 보존과 수사 협조가 필요한 플랫폼·계정·시간 단서
4. 피해자에게 2차 피해를 만들지 않는 증거 제출 범위

## 3. 전문 사례에서 얻은 설계 원칙

Project ReSET 사례는 디지털 성범죄 대응이 단순 삭제요청이 아니라 24시간 모니터링, 채증 자료의 경찰 제공, 피해자·주변인 안내 챗봇, 활동가 워크숍, 경찰 자문단·사이버수사대 협업이 함께 돌아가야 한다는 점을 보여준다.

추적단불꽃의 n번방 공론화 사례는 공개 제보와 관찰 단서라도 시간순으로 축적하고, 사회적 공론화·수사·지원기관 연결이 맞물릴 때 피해 확산을 멈추는 힘이 생긴다는 점을 보여준다.

INTERPOL 피해자 식별 모델은 이미지·영상 분석이 전문 수사관과 승인된 분석가의 협업 영역이라는 점을 강조한다. NIST 디지털 포렌식 지침은 수집, 검사, 분석, 보고 과정에서 증거 무결성과 법률 검토가 중요하다고 본다.

지움AI는 이 원칙을 제품 구조로 번역한다.

- 피해자용: 최초 피해사실 입력, 안전 경고, 증거 장부, 삭제·신고 문안
- 지원자용: 대신 정리하되 원본을 공유하지 않는 보조 모드
- 담당자용: 다이어그램, 증거 지문, 로그 보존 메모, 공식기관 제출 패킷
- 학습용: 피해물 원본 없이 패턴명, 위험도, 근거 ID만 축적

## 4. 엔진 구조

### 4.1 초기피해 접수 엔진

입력값:

- 사건 설명, 긴급 여부, 노출 정보 종류
- URL, 플랫폼, 게시 위치, 관찰 ID·닉네임
- 발견 일시, 캡처/기록 일시, 기록 방식
- 증거 해시 또는 이미지·영상 지문 출처
- 요청 이력, 접수번호, 삭제·재노출 상태

금지값:

- 비밀번호, 주민등록번호, 신분증 원본
- 성적 피해물 원본 파일
- 초대코드, 방 내부 자료, 결제·잠입 방법

### 4.2 리서치·매칭 엔진

목표는 자동 잠입이 아니라 “어떤 단서를 어떤 권한으로 확인해야 하는지”를 나누는 것이다.

- 단어·키워드·별칭 매칭: 공개 검색, 검색 스니펫, 동일 문구·동일 별칭 반복
- 개인정보 변형 매칭: 전화번호 일부, 이메일 일부, 닉네임, 검색 캐시
- 이미지·영상 지문 매칭: 해시, pHash, 썸네일 지문, 기관·플랫폼 권한의 유사 이미지 확인
- 플랫폼 로그 보존 단서: URL, 계정, 게시 시각, 신고 접수번호
- 폐쇄형 채널 단서: 텔레그램, 디스코드, 비공개 서버, 유료방, 초대방
- 은닉·거래 단서: 다크웹·onion 주장, 암호화폐 지갑·결제 요구, 유포협박
- 홍보면·유입면 단서: 공개 게시판 티저, SNS 프로필 링크허브, 댓글 모집, 파일공유 미리보기, 검색 스니펫, 플랫폼 이동 문구

권한 수준:

- VICTIM_SAFE: 피해자가 직접 공개 화면에서 확인 가능
- SUPPORTER_SAFE: 지원자가 피해자 대신 정리할 수 있으나 원본 공유 금지
- OFFICIAL_ONLY: 수사기관·법원 절차가 필요한 로그, IP, 결제, 폐쇄형 채널
- SPECIALIST_ONLY: 이미지·영상 해시 매칭, 포렌식, 전문기관 삭제지원

### 4.3 추격 다이어그램 엔진

노드:

- 사건, 증거, 플랫폼, 관찰 ID, 인프라 신호, 공식 인계

엣지:

- 접근경로, 게시 장소, 관찰된 게시자, 다음 발견, 동일 ID 반복, 패턴 신호, 기관 인계

표현 원칙:

- “관찰”과 “가설”을 분리한다.
- 발견 시각은 최초 유포 시각이 아니라 피해자가 발견한 시각으로 표시한다.
- 게시자 ID는 실명 단정이 아니라 화면에 보인 관찰값으로만 표시한다.

### 4.4 제출 패킷 엔진

기관 제출 패킷에는 다음이 들어간다.

- 사건 요약
- 증거 장부 원문
- 증거 무결성 요약: 메타데이터 지문, 증거 해시, 보강 필요 필드
- 추격 Mermaid 다이어그램
- 리서치·매칭 계획
- 로그 보존·수사 협조 요청 메모
- 연결 기관 후보와 준비물
- 안전 경계

## 5. 4시간 해커톤 구현 전략

4시간 안에 “진짜 작동하는 데모”로 보여줄 범위는 다음이다.

1. 피해자가 최초 피해사실과 증거 2~3건을 입력한다.
2. AI가 사건 유형, 위험도, 삭제 가능성을 분류한다.
3. 접근경로 증거 장부가 만들어지고, 발견·캡처 시각과 해시·접수번호를 보강한다.
4. 시간순 추격 다이어그램이 생성된다.
5. 텔레그램·디스코드·다크웹·암호화폐·재유포 같은 신호가 “피해자 직접 추적 금지, 공식 인계”로 분류된다.
6. 삭제요청 문안, 경찰 신고 준비서, 형사고소 상담 메모, 기관 제출 패킷을 내려받는다.

이번 해커톤에서 하지 않을 것:

- 실제 불법 사이트 크롤링
- 비밀방·다크웹 접속
- 피해물 이미지 업로드 검색
- IP 역추적 또는 가해자 신원 단정
- 자동 기관 제출

## 6. 현재 코드 반영 현황

v2.0 1차 보강에서 반영한 항목:

- 증거 장부에 캡처/기록 일시, 기록 방식, 증거 해시, 해시 출처, 메타데이터 지문, 요청 이력 필드 추가
- 디지털 범죄 경로 지식에 디스코드 비공개 서버, 암호화폐 결제·거래, 이미지·영상 지문 매칭 신호 추가
- 리서치·매칭 엔진 추가: 공개 확인용 쿼리, 권한별 매칭 채널, 보강 필요 항목, 공식 보존 요청 메모 생성
- 기관 제출 패킷 엔진 추가: 추격 다이어그램, 무결성 요약, 법적 권한 분리, 기관 연결 후보 포함
- 로컬 저장 시 URL·민감정보·요청 이력 마스킹 강화

v2.1 운영형 보강에서 추가 반영한 항목:

- 로컬 브라우저 파일 해시 산출: 파일 원본을 저장하지 않고 SHA-256을 산출한다.
- 이미지 평균 해시 기반 시각 지문: 이미지 파일인 경우 `ahash-*` 지문을 함께 산출한다.
- 안전 공개검색 커넥터: Google, Naver, Bing 검색 링크를 만들되 검색 결과 수집·자동 방문은 하지 않는다.
- 공식 제출 커넥터: D4U, ECRM, KISA 등 공식 화면으로 이동할 준비 링크를 만들되 자동 제출은 하지 않는다.
- 비식별 학습 저장소: 사건 원문, URL, 계정, 방 이름 없이 사건 유형, 위험도, 경로 신호 ID, 권한 수준 카운트만 로컬 저장한다.
- 담당자용 감사로그: 생성, 저장, 상태 변경, 제출 패킷 복사·다운로드, 학습 기록 저장, 읽기전용 패킷 생성 이벤트를 남긴다.
- 읽기전용 담당자 패킷: 수정 기능과 자동 제출 없이 사건 정리, 제출 패킷, 감사로그를 정적 HTML로 열거나 내려받는다.

v2.2 홍보면 루트 보강에서 추가 반영한 항목:

- 비공개방 유입면·홍보글 루트 지식베이스: 공개 티저글, SNS 프로필·링크허브, 댓글 모집, 결제·가격 신호, 플랫폼 이동, 파일 미리보기, 검색 스니펫을 안전한 표면 유형으로 분류한다.
- 홍보면 탐지 엔진: 피해자가 이미 본 공개 글·프로필·댓글·제보 메시지의 문구를 기준으로 유입면 후보를 찾는다.
- 제출 패킷 보강: 홍보면 후보, 기록할 항목, 공식 인계 트리거, 수집 체크리스트를 포함한다.
- 비식별 학습 보강: 정확 URL·초대링크·방 이름 없이 홍보면 패턴 ID만 저장한다.

v3.0 운영제품 1차 보강에서 추가 반영한 항목:

- 보안 헤더 추가: nosniff, DENY frame, no-referrer, 민감 브라우저 권한 차단, CSP Report-Only.
- 증거 체인·인계 이력 추가: 발견, 캡처, 요청, 패킷 생성 이벤트를 시간순으로 묶고 `JIUM-CHAIN-*` 매니페스트를 생성한다.
- 제출 패킷 운영성 보강: 증거 체인, 보강 필요 필드, 처리 원칙을 화면과 Markdown 내보내기에 포함한다.
- 실제 운영제품 로드맵 문서화: 제출 패키지, 데스크톱 보안 저장소, 기관 협업, 운영 보안, 합법 데이터 피드 단계를 분리한다.

## 7. 장기 로드맵

### Phase 1: 해커톤 MVP

- 로컬 우선 웹앱
- 증거 장부
- 추격 다이어그램
- 리서치·매칭 계획
- 제출 패킷 Markdown 내보내기
- 발표용 피해자 데모 플레이

### Phase 2: 증거 보존 고도화

- 사용자가 직접 가져온 캡처 파일의 해시 계산: 1차 구현 완료
- 이미지 평균 해시 기반 시각 지문: 1차 구현 완료
- 체인 오브 커스터디 기록: 감사로그 1차 구현 완료
- 사건별 감사 로그
- PDF/PPT/ZIP 제출 패키지

### Phase 3: 안전한 탐색 보조

- 공개 웹 검색 링크 연동은 사용자 확인 후 실행: 1차 구현 완료
- 공개 웹 검색 API 연동은 API 키와 약관 검토 후 선택 기능으로 구현
- 검색 결과 제목·스니펫·URL만 수집
- 피해물 원본 미저장
- 폐쇄형·은닉형 신호는 공식 인계로만 표시

### Phase 4: 기관·전문가 협업

- 읽기 전용 담당자 패킷: 정적 HTML 1차 구현 완료
- 전문기관 승인 피드 기반 위험 패턴 업데이트
- 비식별 패턴 통계
- 다기관 접수 상태 추적
