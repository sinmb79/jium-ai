# 지움AI 피해자 초기대응·추격 엔진 마스터플랜 v2.0

작성: 2026-05-31 · 22B Labs · 보스와 줄리아

## 1. 왜 필요한가

지움AI의 목적은 피해자가 범죄 공간을 직접 추적하게 만드는 것이 아니다. 피해자가 처음 알게 된 사실을 안전하게 구조화하고, 사라지기 쉬운 URL·게시 위치·관찰 ID·발견 시각·삭제요청 이력을 담당 기관이 즉시 이해할 수 있는 제출 패킷으로 바꾸는 것이다.

공식 통계는 이 문제의 규모를 보여준다.

- 여성가족부는 2025년 중앙디지털성범죄피해자지원센터가 피해자 10,637명에게 352,103건을 지원했고, 이 중 삭제지원이 318,020건으로 90.3%였다고 밝혔다. 삭제지원 대상은 불법·유해사이트 51.6%, 검색엔진 25.3%, 소셜미디어 13.5% 순으로 나타났다. 신상정보 삭제지원도 100,242건으로 전년 대비 29.1% 증가했다.
- 개인정보보호위원회는 2025년 개인정보 유출 신고가 447건으로 전년 대비 45.6% 증가했고, 해킹 원인이 276건으로 62%를 차지했다고 발표했다.
- 법무부 디지털 성범죄 등 대응 TF 자료는 다크웹, 디스코드 등 폐쇄·은닉 채널과 암호화폐 결합으로 범죄가 음성화되고, 피해자가 민간 삭제대행에 월 50만~300만 원 이상을 부담하는 현실을 지적한다.

따라서 지움AI는 “탐정”처럼 단서를 정리하고 “행정지원자”처럼 제출자료를 만드는 AI여야 한다. 다만 수사권을 흉내 내지 않는다. IP, 가입자 정보, 결제 흐름, 폐쇄형 채널 내부 자료는 수사기관 또는 법원의 적법 절차로 확보되어야 한다.

## 2. 지향점

피해자에게 필요한 것은 세 가지다.

1. 지금 무엇을 해야 하는지: 피해물 원본을 퍼뜨리지 않고 URL, 게시 위치, 관찰 ID, 발견·캡처 시각만 남긴다.
2. 어디까지 퍼졌는지: 공개 검색, 검색 캐시, SNS, 커뮤니티, 파일공유, 폐쇄형 메신저, 다크웹·암호화폐 신호를 권한 수준별로 분리한다.
3. 누구에게 무엇을 제출할지: 중앙디지털성범죄피해자지원센터, 경찰 ECRM, 방송미디어통신심의위원회, KISA, 플랫폼 신고 경로에 맞는 패킷을 만든다.

사건담당자에게 필요한 것은 네 가지다.

1. 관찰된 사실과 가설의 구분
2. 시간순 전파 다이어그램
3. 로그 보존과 수사 협조가 필요한 플랫폼·계정·시간 단서
4. 피해자에게 2차 피해를 만들지 않는 증거 제출 범위

## 3. 전문 사례에서 얻은 설계 원칙

Project ReSET 사례는 디지털 성범죄 대응이 단순 삭제요청이 아니라 24시간 모니터링, 채증 자료의 경찰 제공, 피해자·주변인 안내 챗봇, 활동가 워크숍, 경찰 자문단·사이버수사대 협업이 함께 돌아가야 한다는 점을 보여준다.

추적단불꽃의 n번방 공론화 사례는 공개 제보와 관찰 단서라도 시간순으로 축적하고, 사회적 공론화·수사·지원기관 연결이 맞물릴 때 피해 확산을 멈추는 힘이 생긴다는 점을 보여준다.

INTERPOL 피해자 식별 모델은 이미지·영상 분석이 전문 수사관과 승인된 분석가의 협업 영역이라는 점을 강조한다. NIST 디지털 포렌식 지침은 수집, 검사, 분석, 보고 과정에서 증거 무결성과 법률 검토가 중요하다고 본다.

지움AI는 이 원칙을 제품 구조로 번역한다.

- 피해자용: 최초 피해사실 입력, 안전 경고, 증거 장부, 삭제·신고 문안
- 지원자용: 대신 정리하되 원본을 공유하지 않는 보조 모드
- 담당자용: 다이어그램, 증거 지문, 로그 보존 메모, 공식기관 제출 패킷
- 학습용: 피해물 원본 없이 패턴명, 위험도, 근거 ID만 축적

## 4. 엔진 구조

### 4.1 초기피해 접수 엔진

입력값:

- 사건 설명, 긴급 여부, 노출 정보 종류
- URL, 플랫폼, 게시 위치, 관찰 ID·닉네임
- 발견 일시, 캡처/기록 일시, 기록 방식
- 증거 해시 또는 이미지·영상 지문 출처
- 요청 이력, 접수번호, 삭제·재노출 상태

금지값:

- 비밀번호, 주민등록번호, 신분증 원본
- 성적 피해물 원본 파일
- 초대코드, 방 내부 자료, 결제·잠입 방법

### 4.2 리서치·매칭 엔진

목표는 자동 잠입이 아니라 “어떤 단서를 어떤 권한으로 확인해야 하는지”를 나누는 것이다.

- 단어·키워드·별칭 매칭: 공개 검색, 검색 스니펫, 동일 문구·동일 별칭 반복
- 개인정보 변형 매칭: 전화번호 일부, 이메일 일부, 닉네임, 검색 캐시
- 이미지·영상 지문 매칭: 해시, pHash, 썸네일 지문, 기관·플랫폼 권한의 유사 이미지 확인
- 플랫폼 로그 보존 단서: URL, 계정, 게시 시각, 신고 접수번호
- 폐쇄형 채널 단서: 텔레그램, 디스코드, 비공개 서버, 유료방, 초대방
- 은닉·거래 단서: 다크웹·onion 주장, 암호화폐 지갑·결제 요구, 유포협박
- 홍보면·유입면 단서: 공개 게시판 티저, SNS 프로필 링크허브, 댓글 모집, 파일공유 미리보기, 검색 스니펫, 플랫폼 이동 문구

권한 수준:

- VICTIM_SAFE: 피해자가 직접 공개 화면에서 확인 가능
- SUPPORTER_SAFE: 지원자가 피해자 대신 정리할 수 있으나 원본 공유 금지
- OFFICIAL_ONLY: 수사기관·법원 절차가 필요한 로그, IP, 결제, 폐쇄형 채널
- SPECIALIST_ONLY: 이미지·영상 해시 매칭, 포렌식, 전문기관 삭제지원

### 4.3 추격 다이어그램 엔진

노드:

- 사건, 증거, 플랫폼, 관찰 ID, 인프라 신호, 공식 인계

엣지:

- 접근경로, 게시 장소, 관찰된 게시자, 다음 발견, 동일 ID 반복, 패턴 신호, 기관 인계

표현 원칙:

- “관찰”과 “가설”을 분리한다.
- 발견 시각은 최초 유포 시각이 아니라 피해자가 발견한 시각으로 표시한다.
- 게시자 ID는 실명 단정이 아니라 화면에 보인 관찰값으로만 표시한다.

### 4.4 제출 패킷 엔진

기관 제출 패킷에는 다음이 들어간다.

- 사건 요약
- 증거 장부 원문
- 증거 무결성 요약: 메타데이터 지문, 증거 해시, 보강 필요 필드
- 추격 Mermaid 다이어그램
- 리서치·매칭 계획
- 로그 보존·수사 협조 요청 메모
- 연결 기관 후보와 준비물
- 안전 경계

## 5. 4시간 해커톤 구현 전략

4시간 안에 “진짜 작동하는 데모”로 보여줄 범위는 다음이다.

1. 피해자가 최초 피해사실과 증거 2~3건을 입력한다.
2. AI가 사건 유형, 위험도, 삭제 가능성을 분류한다.
3. 접근경로 증거 장부가 만들어지고, 발견·캡처 시각과 해시·접수번호를 보강한다.
4. 시간순 추격 다이어그램이 생성된다.
5. 텔레그램·디스코드·다크웹·암호화폐·재유포 같은 신호가 “피해자 직접 추적 금지, 공식 인계”로 분류된다.
6. 삭제요청 문안, 경찰 신고 준비서, 형사고소 상담 메모, 기관 제출 패킷을 내려받는다.

이번 해커톤에서 하지 않을 것:

- 실제 불법 사이트 크롤링
- 비밀방·다크웹 접속
- 피해물 이미지 업로드 검색
- IP 역추적 또는 가해자 신원 단정
- 자동 기관 제출

## 6. 현재 코드 반영 현황

v2.0 1차 보강에서 반영한 항목:

- 증거 장부에 캡처/기록 일시, 기록 방식, 증거 해시, 해시 출처, 메타데이터 지문, 요청 이력 필드 추가
- 디지털 범죄 경로 지식에 디스코드 비공개 서버, 암호화폐 결제·거래, 이미지·영상 지문 매칭 신호 추가
- 리서치·매칭 엔진 추가: 공개 확인용 쿼리, 권한별 매칭 채널, 보강 필요 항목, 공식 보존 요청 메모 생성
- 기관 제출 패킷 엔진 추가: 추격 다이어그램, 무결성 요약, 법적 권한 분리, 기관 연결 후보 포함
- 로컬 저장 시 URL·민감정보·요청 이력 마스킹 강화

v2.1 운영형 보강에서 추가 반영한 항목:

- 로컬 브라우저 파일 해시 산출: 파일 원본을 저장하지 않고 SHA-256을 산출한다.
- 이미지 평균 해시 기반 시각 지문: 이미지 파일인 경우 `ahash-*` 지문을 함께 산출한다.
- 안전 공개검색 커넥터: Google, Naver, Bing 검색 링크를 만들되 검색 결과 수집·자동 방문은 하지 않는다.
- 공식 제출 커넥터: D4U, ECRM, KISA 등 공식 화면으로 이동할 준비 링크를 만들되 자동 제출은 하지 않는다.
- 비식별 학습 저장소: 사건 원문, URL, 계정, 방 이름 없이 사건 유형, 위험도, 경로 신호 ID, 권한 수준 카운트만 로컬 저장한다.
- 담당자용 감사로그: 생성, 저장, 상태 변경, 제출 패킷 복사·다운로드, 학습 기록 저장, 읽기전용 패킷 생성 이벤트를 남긴다.
- 읽기전용 담당자 패킷: 수정 기능과 자동 제출 없이 사건 정리, 제출 패킷, 감사로그를 정적 HTML로 열거나 내려받는다.

v2.2 홍보면 루트 보강에서 추가 반영한 항목:

- 비공개방 유입면·홍보글 루트 지식베이스: 공개 티저글, SNS 프로필·링크허브, 댓글 모집, 결제·가격 신호, 플랫폼 이동, 파일 미리보기, 검색 스니펫을 안전한 표면 유형으로 분류한다.
- 홍보면 탐지 엔진: 피해자가 이미 본 공개 글·프로필·댓글·제보 메시지의 문구를 기준으로 유입면 후보를 찾는다.
- 제출 패킷 보강: 홍보면 후보, 기록할 항목, 공식 인계 트리거, 수집 체크리스트를 포함한다.
- 비식별 학습 보강: 정확 URL·초대링크·방 이름 없이 홍보면 패턴 ID만 저장한다.

v3.0 운영제품 1차 보강에서 추가 반영한 항목:

- 보안 헤더 추가: nosniff, DENY frame, no-referrer, 민감 브라우저 권한 차단, CSP Report-Only.
- 증거 체인·인계 이력 추가: 발견, 캡처, 요청, 패킷 생성 이벤트를 시간순으로 묶고 `JIUM-CHAIN-*` 매니페스트를 생성한다.
- 제출 패킷 운영성 보강: 증거 체인, 보강 필요 필드, 처리 원칙을 화면과 Markdown 내보내기에 포함한다.
- 실제 운영제품 로드맵 문서화: 제출 패키지, 데스크톱 보안 저장소, 기관 협업, 운영 보안, 합법 데이터 피드 단계를 분리한다.

v3.1 보관·이동성 보강에서 추가 반영한 항목:

- 암호화 사건 파일 `.jiumcase` 내보내기: 보관함 사건을 패스프레이즈 기반 암호화 파일로 백업한다.
- 암호화 사건 파일 가져오기: 같은 패스프레이즈로 복호화한 사건을 현재 보관함에 병합한다.
- 파일 평문 노출 방지: `.jiumcase` JSON에는 사건 제목, 정확 URL, 게시 단서가 평문으로 들어가지 않도록 테스트로 고정한다.

v3.2 제출 패키지 보강에서 추가 반영한 항목:

- 기관 제출 ZIP 패키지: Markdown 제출본, 담당자 HTML, 인쇄용 HTML, 증거 체인 manifest JSON, Mermaid 다이어그램, 체크리스트를 한 번에 내려받는다.
- 인쇄용 HTML 제출본: 브라우저 인쇄/저장으로 PDF 제출본을 만들 수 있는 HTML을 제공한다.
- ZIP 생성 로직: 외부 라이브러리 없이 저장 방식 ZIP을 생성하고 파일명과 manifest 포함 여부를 테스트한다.

v3.3 제출 버전 비교 보강에서 추가 반영한 항목:

- 제출 패킷 스냅샷: 패킷 지문, 증거 체인, 증거 ID, 기관 후보, 홍보면 패턴, 보강 항목을 저장한다.
- 직전 버전 비교: 증거·기관·보강 항목 변경을 사용자에게 표시하고 Markdown으로 내려받을 수 있다.
- 민감 원문 최소화: 스냅샷에는 원문 URL, 피해물 원본, 방 이름, 게시자 원문 텍스트를 저장하지 않는다.

v3.4 기관별 제출 워크플로 보강에서 추가 반영한 항목:

- 기관별 제출 프로필: D4U, ECRM, KISA, 방송미디어통신심의위원회, 온라인피해365센터, 대한법률구조공단의 사용 시점, 준비물, 제출 순서, 제출 금지 항목을 구조화한다.
- 제출 준비도 점수: 피해사실 요약, 피해 유형, URL·게시 위치, 발견·캡처 시각, 기록 방식, 증거 지문·해시, 요청 이력 기준으로 기관별 준비도를 표시한다.
- 공식권한 분리: IP, 가입자 정보, 서버 로그, 결제·암호화폐 흐름, 폐쇄형 채널 확인은 피해자 직접 추적이 아니라 수사·심의기관 요청사항으로 남긴다.
- ZIP 제출 패키지 보강: 기관별 워크플로 JSON과 제출 체크리스트 TXT를 포함한다.

v3.5 암호화 보관함 세션 보안 보강에서 추가 반영한 항목:

- 자동 잠금: 암호화 보관함을 열어둔 뒤 5분 동안 활동이 없으면 복호화 목록과 패스프레이즈 입력값을 화면 상태에서 지운다.
- 숨김 탭 잠금: 브라우저 탭이 숨겨지면 즉시 보관함을 잠가 화면공유, 공용 PC, 브라우저 전환 중 노출을 줄인다.
- 수동 잠금: 피해자가 제출자료 확인 후 `지금 잠금`으로 복호화 상태를 바로 정리할 수 있다.
- 상태 표시: 잠김/열림 상태와 자동 잠금까지 남은 시간을 표시해 사용자가 보관함 위험 상태를 이해할 수 있게 한다.

v3.6 운영 보안 게이트 보강에서 추가 반영한 항목:

- dependency audit: 중간 이상 취약 의존성이 발견되면 운영 배포 전에 실패하도록 한다.
- PR 품질 게이트: Pull Request에서 secret scan, audit, typecheck, test, build를 자동 검증한다.
- Pages 배포 게이트: main 배포 전에도 같은 보안·품질 검사를 통과해야 정적 산출물 업로드와 배포가 진행된다.

v3.7 CSP Enforcement·정적 호스팅 헤더 보강에서 추가 반영한 항목:

- CSP Enforcement: `Content-Security-Policy`를 실제 차단 헤더로 추가하고, 같은 정책을 Report-Only에도 유지한다.
- 헤더 중앙화: Next 런타임 헤더와 정적 `_headers` 파일이 같은 보안 헤더 정의를 사용한다.
- 정적 호스팅 준비: Netlify와 Cloudflare Pages가 읽을 수 있는 `public/_headers`를 생성하고 동기화 테스트로 고정한다.
- 운영 한계: GitHub Pages는 `_headers`를 응답 헤더로 강제하지 않으므로, 강제 보안 헤더 운영은 헤더 지원 호스팅으로 이전해야 한다.

v3.8 GitHub Actions Node 24 네이티브 전환에서 추가 반영한 항목:

- 강제 플래그 제거: 런타임 강제 환경변수 대신 공식 action 버전으로 해결한다.
- 운영 지속성: PR 품질 게이트와 Pages 배포 게이트가 같은 Node 24 기반 검증 흐름을 사용한다.

v3.9 운영 URL 보안 헤더 감사에서 추가 반영한 항목:

- 실제 응답 검증: `npm run security:headers:check -- <url>`로 운영 URL의 보안 헤더 적용 여부를 확인한다.
- 호스팅 비교: `_headers` 파일 배포 여부와 실제 응답 헤더 강제를 분리해, 운영 호스팅 이전 판단에 사용한다.
- 회귀 방지: 보안 헤더 감사 테스트를 CI 검증 흐름에 포함한다.

v3.10 담당자 HTML XSS 회귀 게이트에서 추가 반영한 항목:

- HTML 안전 출력: 읽기전용 담당자 패킷과 기관 제출 인쇄본의 사용자 입력 출력을 공통 이스케이프 유틸리티로 처리한다.
- 악성 입력 회귀 테스트: 사건 제목, 설명, URL, 플랫폼, 증거 위치, 게시자 단서, 메모에 들어온 HTML 페이로드가 실행 가능한 태그로 남지 않는지 확인한다.
- CI 게이트: `npm run security:xss`를 PR 품질 게이트와 Pages 배포 게이트에 포함한다.

v3.11 승인 지능 피드 안전 게이트에서 추가 반영한 항목:

- 승인 피드 모델: 기관·파트너·플랫폼 투명성 피드만 수입하고, 출처·확인일·보존기한·권한 수준·감사로그를 필수로 둔다.
- 원문 지표 차단: URL, 초대링크, 계정 핸들, 전화번호, onion 주소는 저장하지 않고, 패턴 ID와 비식별 digest만 허용한다.
- 비식별 학습: 승인 피드도 개별 지표가 아니라 출처 유형, 접근 수준, 경로 패턴, 홍보면 패턴의 집계 통계만 학습 요약으로 사용한다.
- CI 게이트: `npm run security:feeds`를 PR 품질 게이트와 Pages 배포 게이트에 포함한다.

v3.12 제한 피드 운영자 세션·가져오기 UI에서 추가 반영한 항목:

- 운영자 세션: 제한 피드 가져오기와 만료 정리는 짧은 로컬 운영자 세션 안에서만 실행한다.
- 권한 한계 명시: 로컬 세션은 조직 인증·수사권한을 대신하지 않고, 운영 배포 전에는 기관 계정 인증과 서명된 피드 검증이 필요하다.
- 집계 표시: 피해자/대시보드 화면에는 제한 피드 개별 지표명과 원문 지표를 표시하지 않고 패턴별 집계만 보여준다.
- UI 회귀: 제한 피드 가져오기 버튼은 세션 전 비활성화되고, 수입 후에도 개별 지표명이 화면에 노출되지 않는지 테스트한다.

v3.13 서명된 승인 피드 검증에서 추가 반영한 항목:

- signed envelope: 승인 피드는 `jium-authorized-feed-signed-v1` 형식으로 서명 payload와 서명을 분리한다.
- 공개키 pinning: 운영자가 등록한 신뢰 공개키로 RSA-SHA256 서명을 확인한 뒤에만 제한 피드를 수입한다.
- 위조 방지: bundle 내용이 서명 뒤 바뀌거나 미등록/만료 키로 들어오면 수입을 거부한다.
- UI 기본값: 대시보드는 unsigned 제한 피드 수입을 기본 차단하고, 실제 운영 공개키 등록 전에는 가져오기 버튼이 열리지 않는다.

v3.14 신뢰 공개키 레지스트리·등록 검증에서 추가 반영한 항목:

- 공개키 레지스트리: 승인 피드 검증용 공개키를 `data/trusted-authorized-feed-keys.json`으로 분리한다.
- 개인키 유입 차단: 공개키 레지스트리에 private JWK 필드나 private key usage가 섞이면 CI가 실패한다.
- 운영 감사 단서: keyId, issuerName, 유효기간을 구조화해 향후 파트너 키 교체·폐기 절차의 기준으로 삼는다.

v3.15 서명된 운영자 credential 세션에서 추가 반영한 항목:

- 기관/파트너 credential: `jium-authorized-operator-credential-signed-v1`을 공개키로 검증해 제한 피드 운영자 세션을 열 수 있다.
- 가명 운영자 ID: credential subjectId에는 이메일, 전화번호, URL, 초대링크 같은 원문 식별자를 넣지 못하게 한다.
- capability 제한: credential이 부여한 `AUTHORIZED_FEED_IMPORT`, `AUTHORIZED_FEED_SUMMARY`, `AUTHORIZED_FEED_PURGE` 범위 안에서만 세션 권한을 연다.
- 현장 보조 경로: 기존 로컬 확인 문장은 네트워크 없는 현장용 보조 장치로 남기되, 운영 기준은 서명 credential 또는 서버 기반 기관 계정 인증으로 이동한다.

v3.16 서버 기관 계정 RBAC 공통 모델에서 추가 반영한 항목:

- 기관 계정 세션: 서버가 발급할 수 있는 `InstitutionAccountSession` 모델을 추가했다.
- 역할 기반 권한: 피해지원 담당자, 수사기관 연락관, 플랫폼 신뢰·안전 담당자, 프로그램 관리자별 capability matrix를 분리했다.
- MFA 요구: 신뢰 공개키 검토 같은 고위험 권한은 `SERVER_SESSION_MFA`와 `mfaVerifiedAt`을 요구한다.
- 제한 피드 adapter: 서버 기관 계정 세션을 제한 피드 운영자 세션으로 변환해, 향후 실제 로그인 세션이 제한 피드 UI를 열 수 있게 했다.

v3.17 서버 기관 세션 토큰 코어에서 추가 반영한 항목:

- 서버 세션 토큰: `jium-institution-session-token-v1` HMAC 서명 토큰을 발급·검증하는 공통 코어를 추가했다.
- 변조 방지: canonical JSON signing input으로 payload 변조를 검증 실패 처리한다.
- secret 기준: 서버 전용 32바이트 이상 HMAC secret만 허용하고, 약한 secret은 토큰 발급을 거부한다.
- 운영 연결: 실제 API Route를 붙이기 전에도 서버 로그인/세션 발급의 보안 불변식을 테스트할 수 있게 했다.

v3.18 기관 로그인 코어·HttpOnly 쿠키 정책에서 추가 반영한 항목:

- 로그인 코어: signed operator credential을 서버 기관 세션 토큰과 Set-Cookie 헤더로 바꾸는 코어를 추가했다.
- 쿠키 보안: 운영 세션은 `__Host-jium_institution_session` host-only 쿠키로 발급하고, `HttpOnly`, `Secure`, `SameSite=Strict`, `Path=/`를 강제한다.
- 권한 재검증: 서버 역할이 credential capability를 허용하지 않으면 쿠키를 발급하지 않는다.
- 정적 배포 보존: GitHub Pages export를 깨지 않도록 실제 Route 파일 대신 서버 배포에서 호출할 수 있는 코어로 먼저 고정했다.

v3.19 기관 로그인 HTTP 핸들러 코어에서 추가 반영한 항목:

- Route 연결 전 서버 코어: 표준 Web `Request`/`Response` 기반 로그인, 세션 확인, 로그아웃 핸들러를 추가했다.
- 요청 방어: 로그인/로그아웃은 `POST`, JSON content-type, 허용 Origin, `X-Jium-Institution-Login: 1` 헤더, 16KB 본문 제한을 통과해야 한다.
- token 비노출: 성공 응답은 기관 세션 토큰을 JSON에 넣지 않고 HttpOnly cookie로만 전달한다.
- 검증 범위: 세션 쿠키 발급, 세션 확인, 로그아웃 clear cookie, origin/CSRF/body-size/content-type 차단을 테스트한다.

v3.20 기관 인증 비식별 감사 로그에서 추가 반영한 항목:

- 감사 이벤트: 로그인 성공/거부, 세션 확인 성공/거부, 로그아웃 성공/거부 이벤트 모델을 추가했다.
- 데이터 최소화: credential 원문, 세션 토큰, 원문 URL, 초대링크, 계정 핸들, onion 주소, 이메일, 전화번호는 감사 로그에 저장하지 않는다.
- Origin 처리: 실제 Origin 값은 저장하지 않고 허용, 거부, 누락, 미설정으로만 분류한다.
- HTTP 연동: 기관 로그인 HTTP 핸들러가 선택적 audit sink를 통해 비식별 감사 이벤트를 남길 수 있게 했다.

v3.21 기관 인증 감사 해시 체인 원장에서 추가 반영한 항목:

- 해시 체인: 기관 인증 감사 이벤트를 append-only 레코드로 만들고 각 레코드가 이전 recordDigest를 참조하게 했다.
- 변조 탐지: 이벤트 변경, recordDigest 불일치, 이전 digest 연결 오류, sequence 오류를 검증에서 잡는다.
- 운영 연동: 기관 로그인 HTTP 핸들러 audit sink에 바로 연결 가능한 ledger sink를 추가했다.
- 비노출 유지: 원장에도 credential 원문, 서버 세션 토큰, 실제 Origin URL은 저장하지 않는다.

v3.22 서버/데스크톱 감사 원장 JSONL 저장소에서 추가 반영한 항목:

- 영구 저장: 기관 인증 감사 원장을 서버/데스크톱 런타임의 JSONL 파일에 append-only로 저장할 수 있게 했다.
- append 전 검증: 기존 원장이 변조되어 있으면 새 기록 추가를 거부한다.
- 경로 방어: 설정된 base directory 안의 단순 `.jsonl` 파일명만 허용한다.
- 운영 전제: 실제 배포에서는 단일 writer, 백업, 접근 통제, 보존 기간 정책을 함께 적용해야 한다.

v3.23 기관 로그인 Next 서버 Route adapter에서 추가 반영한 항목:

- 서버 연결 준비: login, session, logout을 Next Route Handler에서 호출할 수 있는 adapter로 묶었다.
- 환경 검증: 세션 secret, 허용 Origin, 감사 원장 디렉터리, 신뢰 공개키가 없으면 route 설정을 거부한다.
- secret 방어: `NEXT_PUBLIC_INSTITUTION_SESSION_SECRET`은 설정 자체를 거부하고, 운영 환경에서 insecure cookie도 거부한다.
- 정적 배포 보존: GitHub Pages export를 깨지 않도록 실제 `app/api` 파일은 서버 배포 전환 시점까지 보류한다.

v3.24 배포 프로필 가드에서 추가 반영한 항목:

- 정적 배포 가드: `GITHUB_PAGES=true` 정적 export에서 Route Handler 파일이 들어오면 build 전에 실패한다.
- 서버 운영 가드: `JIUM_SERVER_ROUTES=true` 서버 프로필은 세션 secret, 허용 Origin, 감사 원장 디렉터리를 요구한다.
- 충돌 방지: `GITHUB_PAGES=true`와 `JIUM_SERVER_ROUTES=true`를 동시에 켜면 실패한다.
- CI 연결: GitHub Pages 배포와 PR 품질 게이트 모두 deployment profile guard를 실행한다.

v3.25 서버 Route materialize 흐름에서 추가 반영한 항목:

- 실제 Route 연결: 서버 배포 때 `server-route-templates/app/api/institution/*/route.ts`를 `app/api`로 생성해 login/session/logout Route Handler를 연결한다.
- 정적 배포 보존: GitHub Pages 모드에서는 materialize를 거부하고, 생성 Route 파일은 clean 명령으로 제거한다.
- 덮어쓰기 방지: 사용자가 직접 만든 비생성 Route 파일은 자동으로 덮어쓰지 않는다.
- 검증 연결: materialize/clean, Pages 충돌, 누락 env, 비생성 Route 보호를 회귀 테스트에 포함한다.

v3.26 서버 운영 readiness 가드에서 추가 반영한 항목:

- 운영 승인 분리: 기술적 서버 build와 실제 운영 readiness를 분리해, 공개키 없는 서버 배포가 운영 완료로 오인되지 않게 한다.
- 기관 공개키 조건: 실제 운영 readiness는 최소 1개의 승인 기관 공개키 registry를 요구한다.
- Route 템플릿 조건: login/session/logout 템플릿이 모두 있어야 운영 준비로 판단한다.
- 캐시 정리: 서버 Route clean 시 stale `.next` 타입 캐시도 제거해 정적 검증으로 되돌아갈 수 있게 한다.

v3.27 기관 공개키 승인 패널에서 추가 반영한 항목:

- 공개키 검토: 후보 JWK 공개키에서 private field, private key usage, duplicate keyId, 만료 window 오류를 차단한다.
- 승인 기록: 공개키 fingerprint와 checklist를 산출해 별도 승인 기록에 남길 수 있게 한다.
- registry patch: 앱이 운영 registry를 직접 쓰지 않고, 검토 가능한 JSON patch만 만든다.
- 대시보드 연결: 사건 보드에서 기관 공개키 승인 패널을 열어 서버 readiness 이전의 공개키 등록 절차를 준비한다.

v3.28 기관 감사 원장 검증 패널에서 추가 반영한 항목:

- 원장 검증: 서버/데스크톱 JSONL 감사 원장을 붙여 넣거나 파일로 불러와 해시 체인을 검증한다.
- 운영 리포트: 이벤트 유형, 결과, Origin 분류, 기관명, 최근 기록을 집계해 담당자가 빠르게 확인한다.
- 데이터 최소화: credential 원문, 세션 토큰, URL, 초대링크, 계정 핸들, onion 주소, 이메일, 전화번호는 표시하지 않는다.
- 실패 처리: 검증 실패 원장은 덮어쓰지 않고 원본 보존과 관리자 조사로 분리한다.

v3.29 기관 감사 원장 서버 요약 API에서 추가 반영한 항목:

- 서버 조회 권한: `INSTITUTION_AUDIT_LEDGER_REVIEW` capability를 추가하고 `PROGRAM_ADMIN` MFA 세션에만 허용한다.
- API 요약: `/api/institution/audit-ledger` 서버 Route가 원장 원문 대신 검증 상태, 집계, 최근 비식별 기록만 반환한다.
- 조회 감사: 성공/거부 조회 자체도 감사 원장 이벤트로 남긴다.
- Route materialize: 서버 배포 템플릿과 readiness 검사에 audit-ledger Route를 포함한다.

v3.30 기관 공개키 수명주기·폐기 절차에서 추가 반영한 항목:

- 수명주기 검토: 승인 기관 공개키 registry의 활성, 곧 만료, 만료, 만료일 없음, 활성 전 상태를 판정한다.
- 폐기 절차: 폐기할 keyId와 폐기 시각을 입력해 검토 가능한 retirement patch를 만든다.
- 교체 절차: 새 공개키 승인 후 기존 공개키를 폐기하는 rotation patch 코어를 마련했다.
- 운영 readiness: 서버 운영 준비 검사는 공개키 개수가 아니라 현재 활성 공개키가 1개 이상인지 확인한다.

v3.31 암호화 보관함 보안 저장소 backend 분리에서 추가 반영한 항목:

- backend 분리: 암호화 보관함이 브라우저 localStorage와 데스크톱 보안 저장소 브리지를 같은 API로 사용할 수 있게 했다.
- 데스크톱 연결 계약: 향후 Tauri/Electron 앱은 `window.jiumSecureVault`로 OS 보안 저장소 read/write/delete/describe 기능을 제공한다.
- localStorage 미잔류: 브리지가 연결되면 암호화 payload를 브라우저 localStorage에 남기지 않는다.
- UI 상태 표시: 피해자가 현재 보관 backend와 OS 보안 저장소 연결 여부를 볼 수 있게 했다.

v3.32 기관 계정 관리자 검토·지역 피해지원 라우팅에서 추가 반영한 항목:

- 기관 세션 검토: 운영자가 서버 기관 세션 JSON을 검토해 role, capability, MFA, 만료, 고위험 권한, 원문 식별자 노출 위험을 한 화면에서 확인할 수 있게 했다.
- 감사 리포트: 기관 계정 검토 결과를 Markdown으로 내려받아 운영 전 점검과 승인 기록 보관에 활용할 수 있게 했다.
- 지역 라우팅: 디지털성범죄 사건의 제출 패킷 안에서 지역 디성센터 후보, 중앙디성센터, 1366 긴급 상담 순서를 정리한다.
- 출처 기준: 지역 센터 매핑은 D4U 공식 지역 디성센터 현황 기준으로 표시하고, 제출 전 공식 사이트 재확인을 안전 경계로 둔다.
- 안전 원칙: 지역 라우팅은 자동 제출·자동 접속이 아니라 상담 전 준비 항목과 공식 경로를 정리하는 보조 기능이다.

v3.33 지원자·상담자 읽기전용 전달 패키지에서 추가 반영한 항목:

- 파일 기반 전달: 제출 패킷 안에서 상담자·지원자·수사/심의 담당자에게 줄 `.jiumhandoff.json` 암호화 파일을 만들 수 있게 했다.
- 접근 코드 분리: 파일과 접근 코드를 같은 채널에 남기지 않도록 별도 안내 메모를 함께 생성한다.
- 만료 정책: 전달 파일에는 만료 시각과 수신 역할을 기록하고, 만료된 archive는 복호화 단계에서 거부한다.
- 평문 최소화: archive metadata에는 사건 ID와 체인 지문만 두고, 읽기전용 본문·HTML·증거 manifest는 암호화 payload 안에 넣는다.
- 전달 경계: 지원자는 검토와 질문 정리만 하며, 자동 제출·잠입·IP 추적·신원 단정은 금지 행위로 명시한다.

v3.34 수사·심의기관 제출 전 최종 검수표에서 추가 반영한 항목:

- preflight 판정: 피해 사실 요약, 접근 경로, 시각, 기록 방식, 무결성, 요청 이력, 원본 피해물 취급, 수사권한 분리를 제출 전 점검 항목으로 만들었다.
- 보류/검토 분리: 실제 제출을 막아야 할 BLOCKED 항목과 상담자·기관 확인이 필요한 REVIEW 항목을 분리해 피해자가 무엇부터 보강해야 하는지 볼 수 있게 했다.
- 제출 패킷 통합: 검수표를 제출 패킷 UI와 ZIP 패키지 안에 포함해 담당기관 제출 전 같은 기준으로 다시 확인한다.
- 안전 경계 강화: 비공개방 침투, 초대링크 수집, 계정 구매, 다크웹 접속은 피해자 직접 수행 대상이 아님을 검수표의 안전 경계로 명시했다.

v3.35 데스크톱 OS 보안 저장소 브리지에서 추가 반영한 항목:

- 네이티브 bridge 코어: `scripts/native-secure-vault-bridge.mjs`가 Windows DPAPI, macOS Keychain, Linux Secret Service 저장 경로를 제공한다.
- 기존 UI 재사용: Electron preload가 `window.jiumSecureVault` 계약을 구현하므로 암호화 보관함 UI는 브라우저 localStorage 대신 OS 보안 저장소를 사용할 수 있다.
- 평문 잔류 방지: Windows DPAPI 경로는 사용자 프로필 파일에 DPAPI 보호 blob만 기록하고, 브리지 모드에서는 브라우저 localStorage에 payload를 복사하지 않는다.
- 운영 점검: `npm run desktop:vault:describe`와 `npm run desktop:vault -- write <key> <utf8-file>`, `read|has|delete <key>`로 데스크톱 런타임 연결 상태를 확인할 수 있다.

v3.36 기관 계정 발급·해지 backend에서 추가 반영한 항목:

- 계정 registry: 기관 계정을 가명 subjectId, 역할, capability, 증거 접근 범위, 상태로 관리하고 원문 이메일·전화번호·URL·초대링크를 거부한다.
- 관리자 권한: 계정 발급·해지는 `INSTITUTION_ACCOUNT_ADMIN` capability를 가진 `PROGRAM_ADMIN` MFA 서버 세션만 수행한다.
- 서버 Route: `/api/institution/accounts` 템플릿을 추가하고 CSRF, Origin, Content-Type, body size, HttpOnly 세션 쿠키 검사를 적용한다.
- 감사 기록: 계정 발급, 해지, 조회, 거부 이벤트를 기관 인증 감사 원장에 비식별 필드만으로 기록한다.

v3.37 기관 계정 관리자 UI 서버 연동에서 추가 반영한 항목:

- 서버 운영 UI: 대시보드 기관 계정 관리자 패널에서 서버 계정 목록 조회, 발급, 해지를 수행할 수 있게 했다.
- 쿠키 기반 보안: 계정 관리 요청은 HttpOnly 기관 세션 쿠키와 계정 관리자 보호 header를 전제로 하며, 브라우저에 세션 토큰을 노출하지 않는다.
- 구조화 입력: 기관 ID, 가명 subjectId, role, evidenceAccessScope, 만료시각, capability 목록, 해지 사유 코드를 분리해 감사 가능한 계정 lifecycle로 정리한다.
- 운영 점검 유지: 기존 기관 세션 JSON 검토와 Markdown 리포트 기능은 서버 발급 전후 감사 보조 수단으로 계속 유지한다.

v3.38 기관 계정 승인 workflow에서 추가 반영한 항목:

- 승인자 분리: 계정 발급·해지는 실제 작업자와 다른 승인자 가명 ID를 요구해 단독 처리와 사후 책임 공백을 줄인다.
- 승인 범위 기록: 일반 발급, PROGRAM_ADMIN 발급, 해지를 서로 다른 승인 scope로 저장해 고위험 권한 발급을 별도로 감사할 수 있게 했다.
- 원문 식별자 차단: 승인번호, 승인자 ID, 승인 메모에는 URL, 초대링크, onion 주소, 이메일, 전화번호를 넣지 못하게 했다.
- 현장 감사성: 계정 목록에서 발급승인과 해지승인 번호를 보여주어, 기관 담당자가 내부 승인 문서와 지움AI registry를 빠르게 대조할 수 있게 했다.

v3.39 기관 계정 lifecycle 감사 연결에서 추가 반영한 항목:

- 원장 대조성: 계정 발급·해지 감사 이벤트에 가명 accountId, approvalRef, approvalScope, 대상 역할, 계정 상태를 기록한다.
- 해지 책임성: 계정 해지 감사 이벤트에는 safe reasonCode를 남겨 registry 해지 사유와 감사 원장 기록을 함께 검토할 수 있게 했다.
- 최소 정보 원칙: 감사 원장은 승인자 실명이나 연락처, URL, 초대링크, 메모 원문을 저장하지 않고 내부 승인 참조번호만 남긴다.
- 담당자 handoff: 기관 감사자가 승인 문서, 계정 registry, hash-chain 감사 원장을 같은 기준으로 대조할 수 있게 한다.

v3.40 서버 운영 readiness 리포트에서 추가 반영한 항목:

- 운영 인수인계: 서버 운영 readiness 결과를 JSON 또는 Markdown 리포트로 남길 수 있게 했다.
- 차단 항목 가시화: 서버 profile, 신뢰 공개키, Route template, storage 설정, Origin 설정, deployment guard를 개별 check로 분리한다.
- secret 비노출: 리포트에는 세션 secret 값, Origin 원문, 파일 경로를 넣지 않고 존재 여부와 개수만 표시한다.
- 배포 승인 보조: 기술 readiness 리포트는 실제 기관 승인과 운영 절차 sign-off를 대체하지 않는다는 safety note를 포함한다.
